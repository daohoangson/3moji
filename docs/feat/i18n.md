# Find It! - Internationalization (i18n)

## High-Level Feature Specification

### Overview

Enable cross-language vocabulary learning. A Vietnamese kid opens the app → it detects Vietnamese → shows English words as the learning target with Vietnamese as the support language. The game prompt becomes "Find: dog" while UI chrome, topic names, and hints appear in Vietnamese. The inverse works too: an English-speaking kid can learn Vietnamese words. The experience auto-detects the native language on first visit; the parent can switch both native and target languages as needed.

### Problem

The app currently teaches vocabulary in exactly one language: English. "Find: dog" → tap 🐕. This works for English-speaking kids learning English words, but the architecture has no concept of a **language pair**. There is no separation between "the language the kid is learning" and "the language the UI speaks in."

For a Vietnamese kid, the high-value experience is learning English vocabulary: see "dog", associate it with 🐕, internalize the foreign word. The current app accidentally does this — but the surrounding UI is also in English, so parents can't navigate or understand learning goals. And there's no path for the reverse direction (English kid learning "chó").

### Pain Points

- The word shown in the game IS the learning content. Today it's always English. There is no indirection between the data layer and the display label — the word "dog" is hardcoded in the topic item, not resolved from a translation key.
- UI chrome, topic names, and learning goals are all in English. A Vietnamese parent cannot evaluate whether a topic is age-appropriate for their child.
- No mechanism to express or persist a language pair preference.
- The `<html lang>` tag, metadata, and sitemap are English-only.

### Core Concept: Language Pair, Not Locale

Standard i18n treats language as a single dimension: the user picks a locale and everything renders in that language. This app needs **two independent language axes**:

1. **Native language** — the language the kid (and parent) already knows. Controls UI chrome, topic names, descriptions, learning goals. Detected from browser/geo on first visit.
2. **Target language** — the foreign language being learned. Controls the word shown in "Find: \_\_\_". Defaults to English (the most common second language globally), but switchable.

A Vietnamese kid learning English: native=vi, target=en → UI in Vietnamese, game shows "Find: dog".
An English kid learning Vietnamese: native=en, target=vi → UI in English, game shows "Find: chó".
A Vietnamese kid learning Vietnamese: native=vi, target=vi → everything in Vietnamese (pure native-language mode, equivalent to standard localization).

When native=target, the app collapses to a single-language experience — standard i18n behavior falls out as a special case.

### Analysis Summary

**URL strategy**

The locale prefix encodes the **native language** (the UI/SEO language): `/vi/topics/colors` is the Vietnamese-UI version of the colors topic. The target language is a user preference stored in a cookie, not in the URL — it doesn't affect which "page" you're on, only which word the game displays.

Rationale: a Vietnamese parent searching Google for "trò chơi học tiếng Anh cho bé" (English learning game for kids) should land on `/vi/topics/colors` with Vietnamese UI and English target words. The URL identifies the content language for SEO; the target language is a runtime preference.

|                   | Locale-prefixed URLs                                                                     | Single URL + cookie                                                         |
| ----------------- | ---------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| Structure         | `/vi/topics/colors`, `/en/topics/colors`                                                 | `/topics/colors` (language from cookie/header)                              |
| SEO               | Proper `hreflang` alternates; each language version is independently indexable           | Same URL serves different content; crawlers see one language only           |
| Shareability      | A shared link preserves the recipient's expected language                                | Link language depends on the recipient's own cookie/browser                 |
| Static generation | `generateStaticParams` emits `[locale] × [topicId]` combinations; all pages pre-rendered | One static page per route; language resolved at request time or client-side |
| Migration cost    | All routes move under `app/[locale]/`; existing URLs become the default-locale variant   | Minimal structural change; add middleware + translation calls               |

**Decision: locale-prefixed URLs with `as-needed` prefix mode.** English (default) keeps clean URLs (`/topics/colors`), non-default locales get a prefix (`/vi/topics/colors`). Full SSG preserved.

**Library choice**

`next-intl` for the native language axis — handles routing, middleware, `Accept-Language` detection, cookie persistence, RSC/client translation APIs. ~2KB client bundle.

The target language axis is orthogonal to `next-intl`. It's a simple cookie-based preference (e.g., `TARGET_LOCALE=en`) read in components that render the learning word. No URL impact, no middleware involvement.

**Server/client rendering model**

Middleware-based locale detection does not break static generation. The middleware runs as a lightweight edge function: inspect `Accept-Language` / Vercel's `x-vercel-ip-country` → 307 redirect to the locale-prefixed URL. The destination page is served from static cache. The `[locale]` segment is just another `generateStaticParams` dimension.

What _would_ kill SSG: reading `headers()` or `cookies()` inside a Server Component. The `[locale]` route param avoids this — locale is a route segment, not a runtime lookup.

The target language cookie is read client-side (or via a server action that doesn't affect page rendering), so it also doesn't compromise static generation. The game word resolves at interaction time, not at page render time.

`next-intl` translation APIs map to the existing server/client split:

- **Server Components**: `const t = await getTranslations('namespace')` — async, zero client JS.
- **Client Components**: `const t = useTranslations('namespace')` — hook, requires `NextIntlClientProvider`.

Wired once in `[locale]/layout.tsx`. At our scale (~50 UI keys + ~150 word translations per locale), the full message bundle shipped to the client is negligible.

### Approach Chosen

**Routing**

- Install `next-intl`. Configure locales (initially `en`, `vi`) and default locale (`en`).
- Add `middleware.ts` using `next-intl/middleware`. Detection priority: URL prefix → `NEXT_LOCALE` cookie → `Accept-Language` header → Vercel `x-vercel-ip-country` → default.
- Move all routes under `app/[locale]/`. Root layout sets `<html lang={locale}>`.
- `localePrefix: 'as-needed'` — English URLs unchanged, Vietnamese gets `/vi/`.
- `generateStaticParams` emits `locale × topicId` for full SSG.

**Two-axis language model**

- **Native language** = `[locale]` route param, managed by `next-intl`. Affects UI chrome, topic metadata, SEO, `<html lang>`.
- **Target language** = cookie (`TARGET_LOCALE`), defaults to `en`. Affects only the word shown in game prompts ("Find: \_\_\_") and the success screen label.
- When native=target, the target language concept is invisible — the app behaves like a standard single-language experience.

**Translation structure**

```
messages/
├── en.json    # { "ui": {...}, "topics": {...}, "words": {...} }
└── vi.json
```

Three tiers:

1. **UI chrome** — buttons, labels, headers, error messages (~50 keys).
2. **Topic metadata** — topic names, descriptions, learning goals. Keyed by topic ID.
3. **Words** — the vocabulary items. Keyed by canonical ID (English word). Each locale file contains that language's word for the concept: `en.json` has `"dog": "dog"`, `vi.json` has `"dog": "chó"`.

The game reads the **target language's** word file for the prompt, and the **native language's** word file for hints/support (if shown). Topic data (`lib/topics/data.ts`) keeps English IDs internally; display labels resolve through translation lookup.

**Language switcher**

Minimal controls in the page header:

- Native language: text links (e.g., "EN | VI") using `next-intl`'s `<Link locale="...">`. Navigates to prefixed URL, sets `NEXT_LOCALE` cookie.
- Target language: a separate toggle (e.g., "Learning: English | Tiếng Việt") that sets the `TARGET_LOCALE` cookie. No navigation — just a client-side preference update.

**SEO**

- `<link rel="alternate" hreflang="...">` per native locale.
- Sitemap covers `locale × route` combinations.
- Metadata (title, description) from message files. The Vietnamese sitemap entry for `/vi/topics/colors` has Vietnamese title/description — targeting parents searching in Vietnamese for educational apps.

**`/find/[word]` route**

Becomes `/[locale]/find/[word]`. The word in the URL is user-supplied and language-agnostic. UI chrome translates via native locale. The LLM already accepts any-language input.

### Non-Goals

- More than two locales initially. Architecture supports arbitrary locales; only `en` and `vi` ship first.
- More than two target languages initially. Same — only English and Vietnamese as learning targets.
- Translating the 3,900-entry emoji database (`lib/emoji-data.ts`). Not user-facing as text.
- RTL layout support. Neither English nor Vietnamese requires it.
- Per-locale URL slugs for `/find/[word]` (e.g., `/find/dog` → `/find/chó`). The word is opaque input.
- Translation management UI or CMS. JSON files suffice at this scale.
- Bilingual display in the game (showing both native and target words simultaneously). The game shows only the target word — the kid learns by association with the emoji, not by reading a translation. This can be revisited if user research suggests otherwise.
