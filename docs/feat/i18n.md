# Find It! - Internationalization (i18n)

## High-Level Feature Specification

### Overview

Add multi-language support so the app automatically renders in the user's preferred language. The experience should feel invisible: a Vietnamese parent opening the app on a Vietnamese-configured browser sees Vietnamese immediately, with no language-selection gate. A manual override persists across sessions via cookie.

### Problem

All user-facing text is hardcoded in English: UI chrome, topic names, learning goals, and — critically — the educational words themselves ("dog", "cat", "red"). The app's core value is teaching kids vocabulary through visual association. Without translated words, a Vietnamese toddler is learning English labels, not Vietnamese ones. The app is unusable as a native-language learning tool for non-English speakers.

### Pain Points

- The `<html lang="en">` tag and all metadata are English-only; search engines cannot surface localized results.
- Topic item words (the actual learning content) are untranslatable — there is no indirection layer between the data and the rendered label.
- No locale detection or persistence mechanism exists.
- The `/find/[word]` route accepts freeform input and already passes it to the LLM in any language, but the surrounding UI remains English.

### Analysis Summary

**URL strategy: locale prefix vs. single canonical URL**

Two viable approaches exist for serving localized content:

|                   | Locale-prefixed URLs                                                                     | Single URL + cookie                                                         |
| ----------------- | ---------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| Structure         | `/vi/topics/colors`, `/en/topics/colors`                                                 | `/topics/colors` (language from cookie/header)                              |
| SEO               | Proper `hreflang` alternates; each language version is independently indexable           | Same URL serves different content; crawlers see one language only           |
| Shareability      | A shared link preserves the recipient's expected language                                | Link language depends on the recipient's own cookie/browser                 |
| Static generation | `generateStaticParams` emits `[locale] × [topicId]` combinations; all pages pre-rendered | One static page per route; language resolved at request time or client-side |
| Migration cost    | All routes move under `app/[locale]/`; existing URLs become the default-locale variant   | Minimal structural change; add middleware + translation calls               |

**Recommendation: locale-prefixed URLs with `as-needed` prefix mode.** The default locale (English) keeps clean URLs (`/topics/colors`), while non-default locales get an explicit prefix (`/vi/topics/colors`). This gives us proper SEO for each language, shareable locale-aware links, and full static generation — without polluting English URLs.

The topic pages feature spec explicitly calls out stable URLs and indexability as primary design goals. Locale prefixes extend that strategy to multilingual search.

**Library choice**

`next-intl` is the de facto standard for App Router i18n (~2KB client bundle, full RSC support, built-in middleware for `Accept-Language` negotiation + cookie persistence). The DIY alternative (manual middleware + JSON dictionaries) duplicates what `next-intl` provides out of the box. No reason to hand-roll it.

### Approach Chosen

**Routing**

- Install `next-intl`. Configure locales (initially `en`, `vi`) and default locale (`en`) in a shared routing config.
- Add `middleware.ts` using `next-intl/middleware`. Detection priority: URL prefix → `NEXT_LOCALE` cookie → `Accept-Language` header → default.
- Move all routes under `app/[locale]/`. The root layout reads the locale param and sets `<html lang={locale}>`.
- `localePrefix: 'as-needed'` — English URLs stay as-is, Vietnamese gets `/vi/` prefix.
- `generateStaticParams` in topic pages emits all `locale × topicId` combinations for full SSG.

**Translation structure**

Message files live in `messages/{locale}.json`. Three tiers of content:

1. **UI chrome** — buttons, labels, headers, error messages (~50 keys). Standard `next-intl` message keys.
2. **Topic metadata** — topic names, descriptions, learning goals. Keyed by topic ID within the message file.
3. **Educational words** — the `word` field in topic items. This is the core learning content and must be translated per locale. Keyed by the English canonical word (which doubles as the internal identifier).

Topic data (`lib/topics/data.ts`) keeps using English identifiers internally. The rendering layer resolves display labels through the translation function. Emojis and colors are language-neutral and need no translation.

```
messages/
├── en.json    # { "ui": {...}, "topics": {...}, "words": {...} }
└── vi.json
```

**Language switcher**

A minimal locale toggle in the page header (e.g., "EN | VI"). Uses `next-intl`'s `<Link locale="vi">` which navigates to the prefixed URL and sets the `NEXT_LOCALE` cookie. No modal, no dropdown, no flags — just text links.

**SEO**

- Root layout emits `<link rel="alternate" hreflang="...">` for each locale.
- Sitemap generates entries for all `locale × route` combinations.
- Metadata (title, description) pulled from message files per locale.

**`/find/[word]` route**

This route already accepts any-language input via the LLM. Under i18n routing it becomes `/[locale]/find/[word]`. The UI chrome around it (header, error messages, "Play Again" button) gets translated. The word itself remains user-supplied and language-agnostic.

### Non-Goals

- Supporting more than two locales in the initial implementation. The architecture supports arbitrary locales, but only `en` and `vi` ship first.
- Translating the 3,900-entry emoji database (`lib/emoji-data.ts`). This is used for local content generation fallback and is not user-facing as text.
- RTL layout support. Neither English nor Vietnamese requires it.
- Per-locale routing for `/find/[word]` word slugs (e.g., translating `/find/dog` to `/find/cho`). The word in the URL is an opaque input, not a translatable slug.
- Building a translation management UI or CMS integration. JSON files are sufficient at this scale.
